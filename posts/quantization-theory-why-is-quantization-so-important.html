Quantization Theory - Why is quantization so Important?

The math and engineering behind LLM quantization, from first principles to LLM.int8(), GPTQ, and AWQ.

Every LLM you have ever run locally was probably quantized. You downloaded a Q4_K_M GGUF, loaded it, and it worked. But what actually happened to those weights? What information was destroyed, and why does the model still function?

This post covers the math and engineering behind LLM quantization, from basic theory to three practical methods: LLM.int8(), GPTQ, and AWQ. If you deploy models on-premises or want to fit a 70B model onto fewer GPUs, this is for you.

01 Foundation

What Quantization Actually Is

A neural network is a collection of floating-point numbers. A 70B parameter model in FP16 (16 bits per weight) occupies 140 GB of memory. That is more than a single GPU can hold. 

q = round((x - zero_point) / scale)
scale = (max - min) / (2^bits - 1)
x_hat = q * scale + zero_point

Using x_hat instead of x introduces quantization error. The entire problem is balancing that error against memory and speed gains. If you know signal processing, this is the same analog-to-digital tradeoff: lower bit depth means more quantization noise.

The Memory Math

FP32: 4 bytes/param  -> 70B model = 280 GB
FP16: 2 bytes/param  -> 70B model = 140 GB
INT8: 1 byte/param   -> 70B model = 70 GB
INT4: 0.5 byte/param -> 70B model = 35 GB

Moving from FP16 to INT4 can be the difference between needing two high-end GPUs and fitting on one. It also improves throughput, because inference is usually memory-bandwidth-bound.

02 The Breakthrough

LLM.int8() and the Outlier Problem

Naive INT8 rounding works for smaller models, but breaks at larger scales. The reason is activation outliers: a tiny set of channels has very large values while most channels remain near zero.

If scale must cover rare values near 100 while most data lies in [-3, 3], you lose precision where the model needs it most.

LLM.int8() solves this with mixed precision decomposition: outlier channels stay in FP16, the rest run in INT8, and results are summed. The FP16 path is tiny in cost but preserves quality.

03 Second-Order Methods

GPTQ: Quantize, Then Compensate

GPTQ targets INT4 quantization where only 16 values per weight exist. It uses second-order information (Hessian approximation) to redistribute quantization error across still-unquantized weights. 

Goal: minimize ||W X - W_hat X||^2
H = 2 X X^T

Instead of accepting rounding damage, GPTQ compensates it with a closed-form update derived from Optimal Brain Surgeon ideas, then makes the method practical with fixed ordering, batched lazy updates, and Cholesky-based inversion.

04 Activation-Aware

AWQ: Protect Before You Quantize

AWQ asks a different question: can we protect the important weights first, then quantize? Weight importance is estimated from activation magnitude. Channels with larger activations are more error-sensitive. 

Y = W X = (W * diag(s)) * (diag(s)^-1 * X)
s_j = mean(|x_j|)^alpha, alpha ~= 0.5

This keeps output mathematically equivalent while reshaping weight distributions so important channels use quantization range more effectively. Result: strong INT4 quality with hardware-friendly uniform quantization.

05 GPTQ vs AWQ

GPTQ is second-order and highly effective, but slower to produce. AWQ is first-order, fast, and often preferred in deployment stacks like vLLM and TensorRT-LLM.

If you use llama.cpp, GGUF variants such as Q4_K_M are common practical choices for consumer hardware.

06 Evaluation

Perplexity is the first metric, but not the only one. INT8 usually has negligible loss, while INT4 can introduce measurable degradation, especially in smaller models. Always validate on your real downstream tasks.

07 Deployment Framework

For 70B-class models: FP16 if you have enough VRAM, INT8 around 80 GB VRAM, INT4 for 40-48 GB VRAM, and GGUF INT4 for consumer setups. Quantization is lossy compression. The right choice depends on model size, workload, and hardware budget.
